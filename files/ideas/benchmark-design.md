# whyso 정량 벤치마크 실험 설계

## 목표

whyso가 AI 에이전트의 코드베이스 탐색 효율을 실제로 개선하는지 정량 측정한다.

## 실험 구조

### 환경

- 같은 머신, 같은 프로젝트, 두 개 브랜치
- 브랜치 A (대조군): CLAUDE.md에 whyso 관련 내용 없음
- 브랜치 B (실험군): CLAUDE.md에 whyso 사용법 포함 + `.whyso/_map.md`, `.whyso/*.yaml` 캐시 제공
- 순차 실행, 매 실행마다 세션 초기화

### 프로젝트 준비

1. 신규 프로젝트를 Claude Code로 바이브 코딩 (중규모, 파일 30~50개)
2. 충분한 대화 세션 축적 (히스토리가 의미를 가질 정도)
3. 두 브랜치에 동일하게 복제
4. 실험군 브랜치에만 whyso 설치 및 `.whyso/` 캐시 생성

### 태스크 설계

유형별 3~5종류, 난이도 균일하게.

| 유형 | 예시 태스크 |
|---|---|
| 탐색 | "사용자 인증 로직이 어디에 있는지 찾아서 설명해라" |
| 수정 | "주문 생성 API에 할인 코드 필드를 추가해라" |
| 디버그 | "주문 상태 전이에서 cancelled → refunded가 안 되는 이유를 찾아라" |
| 리팩토링 | "이메일 발송 로직을 별도 서비스로 분리해라" |
| 크로스컷 | "결제 관련 모든 파일을 찾아서 변경 히스토리를 요약해라" |

### 반복 횟수

- 태스크당 최소 5회 반복 (비결정성 보완)
- 총 실행: 5 태스크 × 5회 × 2그룹 = 50회

## 측정 지표

세션 JSONL에서 자동 추출 가능.

### 1차 지표 (핵심)

| 지표 | 정의 | 추출 방법 |
|---|---|---|
| Grep 호출 수 | 코드 탐색에 사용한 Grep/Glob 횟수 | tool_use에서 Grep/Glob 카운트 |
| 첫 Grep 적중률 | 첫 Grep이 원하는 코드를 찾은 비율 | 첫 Grep 직후 Edit/Write가 따라오는지 |
| Read 호출 수 | 파일 읽기 횟수 | tool_use에서 Read 카운트 |
| 총 tool_use 수 | 태스크 완료까지 전체 도구 호출 수 | tool_use 전체 카운트 |
| 총 토큰 수 | 태스크 완료까지 소비한 토큰 | JSONL 메시지 길이 합산 |

### 2차 지표 (보조)

| 지표 | 정의 |
|---|---|
| 태스크 성공률 | 정확하게 완료했는지 (수동 판정) |
| 완료 시간 | 첫 메시지 ~ 마지막 메시지 시간 차 |
| 헛 Grep 비율 | 결과 0건인 Grep의 비율 |
| Agent 스폰 수 | 서브에이전트 사용 횟수 |

## 데이터 수집 자동화

whyso 자체로 양쪽 세션을 분석할 수 있다.

```bash
# 대조군 세션 분석
whyso changes --sessions-dir <대조군_세션_경로> > control.tsv

# 실험군 세션 분석
whyso changes --sessions-dir <실험군_세션_경로> > experiment.tsv
```

tool_use 유형별 카운트는 JSONL 직접 파싱으로 추출:

```bash
# Grep/Glob/Read/Write/Edit 호출 수 집계
grep '"tool_use"' <session>.jsonl | grep -o '"name":"[^"]*"' | sort | uniq -c
```

## 통계 분석

- 각 지표별 평균, 표준편차 산출
- Mann-Whitney U 검정 (비모수, 소표본에 적합)
- p < 0.05 기준으로 유의성 판단

## 예상 결과

| 지표 | 대조군 예상 | 실험군 예상 |
|---|---|---|
| Grep 호출 수 | 많음 (추측 검색) | 적음 (정확 검색) |
| 첫 Grep 적중률 | 낮음 | 높음 |
| 총 tool_use 수 | 많음 | 적음 |
| 총 토큰 수 | 많음 | 적음 |
| 헛 Grep 비율 | 높음 | 낮음 |

## 주의사항

- Claude Code는 비결정적이므로 단일 실행 결과에 의존하지 않는다
- API rate limit으로 동시 실행 불가 — 순차 실행하되 교차 배치 (A-B-A-B...)
- 태스크 순서 효과 방지: 랜덤화 또는 라틴 스퀘어 설계
- 프로젝트가 너무 작으면 whyso 효과가 미미, 너무 크면 실행 비용 과다
